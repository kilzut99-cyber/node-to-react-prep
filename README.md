## 🛠 Технологический стек и программные решения

### 1. HTML5 (Структурная разметка)
*   **Семантическое проектирование**: Использование тегов `<header>`, `<main>`, `<section>` и `<footer>` для формирования логической иерархии документа и улучшения доступности.
*   **Интерактивные идентификаторы**: Применение уникальных `id` (`searchBtn`, `loader`, `repos`) для обеспечения точной связи между DOM-узлами и логикой JavaScript.
*   **UX-индикация**: Выделение специализированных контейнеров для динамического вывода системных ошибок и визуального индикатора процесса загрузки (Loader).

### 2. CSS3 (Интерфейс и управление состояниями)
*   **Layout**: Реализация гибких сеток на базе **Flexbox** для позиционирования элементов карточки профиля и адаптивного размещения тегов истории поиска.
*   **Декларативный подход**: 
    *   Использование служебных классов (например, `.hidden`) для управления отображением элементов без прямого изменения стилей через JS.
    *   Применение псевдокласса `:disabled` для визуального подтверждения блокировки ввода во время активного сетевого ожидания.
    *   Интерактивные эффекты `:hover` для кнопок-тегов, обеспечивающие визуальный отклик (Feedback).

### 3. JavaScript (Асинхронная логика и безопасность)
*   **Архитектура Event Loop**: Разработка логики с учетом приоритетов очередей задач. Синхронные операции очистки интерфейса выполняются в **Call Stack**, а обработчики сетевых ответов делегируются в **Microtask Queue**, что гарантирует плавность работы интерфейса (Rendering).
*   **Fetch API**: Современный встроенный интерфейс для выполнения фоновых HTTP-запросов к внешним серверам по асинхронной модели.
*   **Async/Await**: Синтаксическая надстройка над промисами, позволяющая писать асинхронный код в императивном стиле, исключая «пирамиду погибели» (Callback Hell).
*   **Promise.all**: Конкурентное выполнение независимых запросов к API (профиля и репозиториев), что минимизирует общее время ожидания (Latancy).
*   **LocalStorage API**: Организация локального хранилища для сохранения истории последних 3-х уникальных поисковых сессий.
*   **Security (XSS Protection)**: Принцип «нулевого доверия» к внешним данным. Использование `textContent` и `createElement` исключает интерпретацию данных от API как исполняемого кода.

## 💬 Ответы на контрольные вопросы (Interview Preparation)

> **1. Чем async/await лучше чистых .then()?**
>
> Использование синтаксиса **async/await** является наиболее современным и предпочтительным методом управления асинхронностью. В отличие от цепочек `.then()`, которые при усложнении логики неизбежно ведут к глубокой вложенности («пирамиде погибели»), `async/await` делает структуру кода «плоской» и линейной. 
> 
> Ключевым преимуществом является возможность использования классической конструкции **try...catch** для обработки исключений. Это позволяет разработчику централизованно обрабатывать как сетевые сбои, так и логические ошибки в одной точке, что значительно упрощает отладку и поддержку приложения.

> **2. Почему fetch не переходит в блок catch при ошибке 404?**
>
> Механика работы **Fetch API** основана на успешном завершении сетевого контракта. Согласно лекции, «курьер» (метод fetch) успешно доставил «конверт» (Response) от сервера по указанному адресу. Статус **404 (Not Found)** — это корректный ответ сервера, сообщающий об отсутствии ресурса, а не о техническом сбое соединения. 
> 
> Промис переходит в состояние **Rejected** (блок `catch`) только в случае невозможности физического выполнения запроса: при отсутствии интернет-соединения, DNS-ошибках или недоступности сервера. Именно поэтому в коде реализована ручная проверка свойства **response.ok** для корректной идентификации бизнес-ошибок API.

> **3. Что такое Promise.all и зачем мы его использовали?**
>
> **Promise.all** — это статический метод, позволяющий агрегировать массив промисов и ожидать их одновременного разрешения. В проекте GitHub Spy получение данных профиля и репозиториев являются независимыми процессами. 
> 
> Если бы мы использовали последовательный `await`, время ожидания пользователя равнялось бы сумме времен двух запросов. Благодаря **Promise.all**, запросы выполняются параллельно (конкурентно), и интерфейс обновляется за время, равное длительности самого долгого из них. Это классический пример оптимизации времени (Time Optimization), критически важный для обеспечения отзывчивости пользовательского интерфейса.
